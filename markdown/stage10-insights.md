# 文件洞察：阶段十（文档编写）

## 洞察目标
让交付文档与当前代码实现完全对齐，避免“需求里提过但项目里没做”或“代码已经变了但文档还停留在旧状态”的偏差。

## 关键文件与发现

1. `docs/deployment.md`
   阶段十的部署文档不能照搬需求里的泛化部署表述，而必须以当前 `docker-compose.yml`、`backend/Dockerfile`、`docker/frontend/Dockerfile`、`docker/nginx/default.conf` 为准。
   真实启动链路是：MySQL 健康检查通过后，后端执行 `prisma migrate deploy`，再执行运行时种子 `dist/scripts/seedRuntime.js`，最后前端 Nginx 对外提供页面并反代 `/api`。
   本次二次修订后，部署文档又进一步切换为“Debian 12 + 1Panel”视角：保留底层 Compose 事实不变，但交付说明改成由 1Panel 负责文件上传、编排创建、网站反向代理与 HTTPS。
   这里最关键的新增洞察是：在 1Panel 场景下，前端容器不能继续直接占用服务器 80 端口，否则会与 OpenResty 冲突，必须先改成内部中转端口，再由网站模块接入。

2. `docs/api.md`
   实际项目不是“计划里的 7 个 API”，而是已经落地为 10 个路由：
   3 个公开接口 + 7 个后台接口。
   这个数量差异必须以 `backend/src/routes/` 为准，否则 API 文档会漏掉后台产品列表、配置查询/更新等真实接口。

3. `docs/admin-guide.md`
   后台手册必须只描述实际可操作页面。
   当前后台有登录、产品 CRUD、公告管理、配置管理和退出登录，但没有“管理员密码修改页”或“已删除产品回收站”，因此手册中应明确这是当前限制，而不是虚构操作路径。

4. `README.md`
   新接手开发者最容易混淆的点是有两套环境变量：
   - 根目录 `.env`：给 Docker Compose 用
   - `backend/.env`：给本地直接启动后端用
   README 已把这两者明确拆开，否则本地开发和容器部署会互相污染。

5. `markdown/progress.md`
   原进度文档在阶段九已经追加完成记录，但前面的阶段九表格仍停留在“待执行”，阶段十也还是空白。
   本次同步时已统一修正为：阶段九、阶段十表格完成状态与尾部追加记录一致，并将总进度更新为 `66 / 66`、`100%`。

6. `markdown/task-checklist.md`
   阶段十四个子任务均已按顺序勾选：
   - T-10.1 部署文档
   - T-10.2 API 文档
   - T-10.3 管理员手册
   - T-10.4 README
   这保证执行清单与实际交付同步闭环。

## 阶段十的关键结论

- 部署文档的可信度取决于是否忠实反映真实 Compose 链路，而不是需求中的理想化部署图。
- 当部署方式从“纯命令行 Docker”切换到“1Panel 面板托管”时，文档必须同时说明端口接入方式随之变化，尤其是 OpenResty 与前端容器的职责边界。
- API 文档必须以路由文件为准，不能只看计划文档。
- 管理员手册要覆盖真实能力，也要明确当前限制。
- 进度文档不仅要追加新记录，还要修正旧摘要，否则会自相矛盾。
