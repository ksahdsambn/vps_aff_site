# VPS AFF引流下单网站需求文档

## 一、项目概述

本项目旨在构建一个VPS产品展示和推广平台，用户可以通过该平台浏览、筛选、排序各类VPS产品，并通过AFF链接进行下单。系统需要支持中英文双语、移动端适配、后台管理等功能。

## 二、技术栈选型

### 2.1 技术栈方案

**前端框架:** React 18 + TypeScript
- 生态成熟，组件化开发模式适合本项目
- TypeScript提供类型安全，减少调试成本

**UI组件库:** Ant Design 5.x
- 开箱即用的企业级组件
- 内置国际化支持
- 表格、筛选、排序功能完善

**后端框架:** Node.js + Express.js + TypeScript
- JavaScript/TypeScript全栈统一
- Express轻量级，易于部署和维护

**数据库:** MySQL 8.0
- 关系型数据库适合结构化VPS产品数据

**ORM框架:** Prisma
- TypeScript原生支持，类型推导能力强
- 数据库迁移管理方便
- 开发体验优秀

**构建工具:** Vite
- 开发体验优秀，构建速度快
- React官方推荐，配置简洁

### 2.2 部署环境

- 操作系统: Debian 12
- 部署方式: Docker Compose
- 采用Docker Compose编排三个服务：MySQL数据库容器、Node.js后端容器、Nginx前端容器
- Docker Compose适合本项目多容器场景，可通过一条命令启动所有服务，统一管理网络和依赖关系

## 三、功能模块设计

### 3.1 前端展示模块

#### 3.1.1 产品列表页面

**页面布局**

- 顶部区域

- 公告栏区域
  - 静态固定位置，显示在页面顶部
  - 支持纯文本或Markdown格式内容
  - 背景色可配置，默认浅黄色提示样式

- 筛选与搜索区域
  - 服务商下拉选择框（支持多选）
  - 产品名称搜索框（模糊搜索）
  - 位置搜索框（模糊搜索）
  - 清空筛选按钮

- 表格区域
  - 列包括：服务商、产品名称、CPU、内存、硬盘、月流量、带宽、位置、价格/年、测评、备注、下单
  - 可排序列：CPU、内存、硬盘、月流量、价格/年、带宽
  - 排序方式：点击列头切换升序/降序/默认
  - 默认每页显示50条数据

**表格列详细说明**

1. 服务商：纯文本显示
2. 产品名称：纯文本显示
3. CPU：数值型，支持按数值大小排序
4. 内存：数值型，支持小数，单位为GB，支持按数值大小排序
5. 硬盘：数值型，支持小数，单位为GB，支持按数值大小排序
6. 月流量：数值型，前端统一显示为TB，保留两位小数，支持按数值大小排序
7. 带宽：数值型，前端统一显示为Gbps，保留两位小数，支持按数值大小排序
8. 位置：纯文本显示（如：洛杉矶、香港等）
9. 价格/年：数值+货币代码，保留两位小数，支持按数值大小排序（不同货币按数值直接比较，前端标注"按数值排序仅供参考"，不进行汇率转换）
10. 测评：超链接文本，点击跳转到测评页面
11. 备注：纯文本显示
12. 下单：超链接按钮，点击跳转到AFF购买链接

3. CPU核心数：数字输入框（必填，整数）
4. 内存：数字输入框，单位固定为GB，支持小数（必填）
5. 硬盘：数字输入框，单位固定为GB，支持小数（必填）
6. 月流量：数字输入框，单位可选GB或TB（必填）
7. 带宽：数字输入框，单位可选Mbps或Gbps（必填）

- 分页数量设置

### 3.3 数据处理模块

#### 3.3.1 单位自动转换逻辑

**月流量转换**

- 后台支持输入单位：GB或TB
- 存储格式：统一转换为GB存储到数据库
- 转换公式：1TB = 1000GB
- 前端显示：统一转换为TB，保留两位小数
- 示例：
  - 输入500GB → 存储500 → 显示0.50TB
  - 输入0.5TB → 存储500 → 显示0.50TB
  - 输入1TB → 存储1000 → 显示1.00TB

**带宽转换**

- 后台支持输入单位：Mbps或Gbps
- 存储格式：统一转换为Mbps存储到数据库
- 转换公式：1Gbps = 1000Mbps
- 前端显示：统一转换为Gbps，保留两位小数
- 示例：
  - 输入500Mbps → 存储500 → 显示0.50Gbps
  - 输入0.5Gbps → 存储500 → 显示0.50Gbps
  - 输入1Gbps → 存储1000 → 显示1.00Gbps

#### 3.3.2 价格处理逻辑

**输入格式**

- 支持格式："数值 货币代码"，如"22 USD"、"49.99 EUR"
- 数值部分精确到小数点后两位
- 货币代码为3位字母（ISO 4217标准）

**存储格式**

- 数值和货币代码分开存储

**显示格式**

- 前端统一显示为："数值 货币代码"
- 示例："22.00 USD"、"49.99 EUR"

#### 3.3.3 排序逻辑

**数值排序**

- CPU、内存、硬盘：按原始数值排序
- 月流量：按GB值排序（即存储的数值）
- 带宽：按Mbps值排序（即存储的数值）
- 价格：按数值部分排序，忽略货币代码，前端标注"按数值排序仅供参考"，不进行汇率转换

**排序方向**

- 默认：不排序（按添加时间或ID）
- 升序：从小到大
- 降序：从大到小
- 点击列头循环切换：默认 → 升序 → 降序 → 默认

## 五、API接口设计

### 5.1 前端API

#### 5.1.1 产品列表接口

- 请求方式：GET
- 路径：/api/products
- 参数：
  - 服务商列表（可选，用于筛选，支持多选）
  - 排序字段（可选）
  - 排序方向（可选，升序或降序）
  - 页码（可选，默认第1页）
  - 每页数量（可选，默认50条）
  - 产品名称关键词（可选，模糊搜索）
  - 位置关键词（可选，模糊搜索）
- 响应：
  - 总记录数
  - 当前页码
  - 每页数量
  - 产品列表

#### 5.1.2 服务商列表接口

- 请求方式：GET
- 路径：/api/providers
- 参数：无
- 响应：去重后的服务商名称列表

#### 5.1.3 系统配置接口

- 请求方式：GET
- 路径：/api/config
- 参数：无
- 响应：
  - 公告内容（中英文）
  - 社交媒体链接（Telegram、YouTube、测评博客、X）
  - 网站标题（中英文）
  - Logo地址

### 5.2 后台管理API

- 请求方式：POST
- 路径：/api/admin/login

#### 5.2.2 产品管理接口

**添加产品**

- 请求头：携带Token令牌

**删除产品**

- 请求方式：DELETE
- 路径：/api/admin/products/{产品ID}
- 请求头：携带Token令牌

**查询产品列表**

- 请求方式：GET
- 路径：/api/admin/products
- 请求头：携带Token令牌
- 参数：搜索关键词、页码、每页数量

#### 5.2.3 配置管理接口

**获取配置**

- 请求方式：GET
- 路径：/api/admin/config
- 请求头：携带Token令牌

**更新配置**

- 请求方式：PUT
- 路径：/api/admin/config
- 请求头：携带Token令牌

## 六、部署方案

### 6.1 Docker Compose部署

#### 6.1.1 环境准备

**服务器要求**

- 操作系统：Debian 12
- 已安装Docker和Docker Compose
- 开放端口：80（HTTP）、443（HTTPS）

#### 6.1.2 Docker Compose编排

本项目采用Docker Compose编排三个服务容器：

- MySQL数据库容器：运行MySQL 8.0，持久化数据存储
- Node.js后端容器：运行Express后端服务
- Nginx前端容器：运行前端静态文件，同时作为反向代理

#### 6.1.3 项目部署步骤

**后端服务配置**

- 配置环境变量，包括：
  - 数据库连接地址（指向Docker内部MySQL容器）
  - 数据库端口
  - 数据库用户名
  - 数据库密码
  - 数据库名称
  - JWT密钥
  - 服务端口
- 构建后端镜像

**前端服务配置**

- 构建前端项目生成静态文件
- 将静态文件打包到Nginx镜像中

**Nginx配置**

- 配置前端网站为主域名
- 配置反向代理，将/api路径代理到后端服务容器
- 启用gzip压缩
- 配置SSL证书（推荐）

#### 6.1.4 启动与维护

**启动服务**

- 通过Docker Compose一条命令启动所有服务
- 所有容器自动连接到同一网络，相互通信

**服务管理**

- 查看服务运行状态
- 查看容器日志
- 重启单个或全部服务
- 停止并销毁所有容器

**数据持久化**

- MySQL数据通过Docker卷挂载到宿主机，确保数据不丢失
- 后端上传文件通过卷挂载持久化

### 6.2 初始化流程

**数据库初始化**

- 运行数据库迁移脚本，创建所有数据表
- 插入默认系统配置数据
- 创建初始管理员账号（用户名admin，密码需修改）

**首次配置**

- 登录后台管理系统

- 配置社交媒体链接

### 7.1 前端安全

- 所有用户输入进行XSS过滤

### 7.2 后端安全

## 八、错误处理规范

### 8.1 统一错误响应格式

所有API接口返回统一的错误响应结构，包含以下字段：

- 错误码：数字类型，用于前端程序判断
- 错误消息：字符串，面向用户的可读信息

### 8.2 错误码分类

**通用错误码**

- 400：请求参数错误
- 401：未登录或Token过期
- 403：无权限访问
- 404：资源不存在
- 500：服务器内部错误

**业务错误码**

- 1001：用户名或密码错误
- 1002：登录失败次数过多，请稍后再试
- 2001：产品不存在
- 3001：配置项不存在
- 3002：配置更新失败

### 8.3 前端错误处理

- 网络请求失败时显示友好提示
- 表单校验错误时定位到具体字段并显示错误信息
- Token过期时自动跳转到登录页面
- 服务器错误时显示通用错误页面

### 8.4 后端错误处理

- 所有异常统一捕获，避免程序崩溃
- 记录错误日志，便于排查问题
- 生产环境不暴露详细错误堆栈给前端
- 数据库操作失败时进行回滚处理
